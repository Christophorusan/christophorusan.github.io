Oslo

Oslo is an AI workflow system for crypto GTM. It reads market, BD, product, and team signals, routes them through shared memory, then turns them into daily briefs, ranked next moves, campaign drafts, and approval-gated actions.

Surface Discord, Telegram, CLI, portfolio
Units BD, marketing, ops, data, engineering, security
Rule Propose first. Humans approve.

What it is

Most company knowledge sits in the worst possible places: Telegram threads, Discord channels, half-updated docs, founder memory, scattered GitHub issues, screenshots, and a few dashboards no one checks at the right moment.

Oslo turns that mess into a repeatable workflow. It reads signals, assembles context, chooses the right unit, drafts a recommendation, and posts it where the team already works. It is not an autonomous employee. It is a way to make the next useful move visible.

Daily workflow

The strongest version of Oslo is not "ask a chatbot." It is a daily operating rhythm where every unit starts from the same context and every proposed move has evidence.

Morning briefPulls open proposals, market state, top BD target, content gap, ops issue, and one self-improvement.
08:30
Context of the dayWrites the shared daily context every downstream agent reads before drafting anything.
09:00
Unit passesBD suggests contacts, marketing drafts content, data surfaces market signals, ops ranks what matters.
HITL
Approval queueEvery external action is a proposal: approve, edit, reject, or postpone. Nothing important auto-sends.
Weekly

Command shape

Each skill has typed inputs, a predictable output shape, and an explicit human-in-the-loop flag. The point is not to let agents improvise. The point is to make useful work repeatable. 

/oslo hot
/oslo note Selini "Asked about kHYPE vault docs"
/oslo suggest-next-moves --motion=convert
/oslo daily
/oslo voice-check draft.md

In production, external actions route through proposals. Oslo can draft. The operator decides.

Why it matters

The value is not "AI wrote a post." The value is that the post knows the same thing the BD agent knows, the BD suggestion knows what marketing shipped, and the weekly brief knows which actions produced replies, meetings, deposits, or silence.

  • Less context loss: the same memory follows work across Discord, Telegram, docs, and code.
  • Better prioritization: accounts and moves are ranked by evidence, not whoever was most recently discussed.
  • Faster learning: accepted suggestions can be watched for downstream outcomes.
  • Safer execution: the system proposes drafts and next moves, while humans approve external actions.

Stress tests

A workflow system only matters if it behaves when the input stream is ugly: missing data, stale memory, duplicate contacts, contradictory signals, and too many proposals.

Noisy signals

Duplicate accounts

Same partner appears through X, Telegram, and notes. Oslo should merge context, not create three priorities.

Stale state

Decay without touch

A hot lead should cool if nothing happens. Silence is a signal, not an absence of data.

Bad drafts

Voice check

Marketing and BD drafts run through a style gate before they reach the team for approval.

Cost pressure

Usage ledger

Every run should log tokens, latency, cost, and output class so expensive agents justify themselves.

Roadmap

The next version of Oslo is less about adding more agents and more about closing the loop between suggestion and outcome.

  • GTM recommendation layer: top retain risks, top convert moves, and an 8-hour allocation screen.
  • Outcome attribution: track replies, meetings, shipped posts, deposits, churn, and no-response windows.
  • Context of the day: one shared struct that every downstream unit reads before drafting.
  • Knowledge refresh: weekly proposed patches to unit knowledge files from canonical sources.
  • Event-driven mode: named-account watchlists that surface opportunities within minutes.
